# CLAUDE.md

LLM이 코딩할 때 자주 저지르는 실수를 줄이기 위한 행동 지침이다.
프로젝트별 지침이 있다면 이 문서와 함께 병합해서 사용한다.

주의:
이 지침은 속도보다 신중함에 더 무게를 둔다.
아주 단순한 작업에서는 상황에 맞게 판단한다.

---

## 0. 이 프로젝트의 최우선 규칙 — 설계문서 먼저, 상의 먼저 (VibeNews)

**모든 코딩/기능 구현 전에 예외 없이 다음 순서를 지킨다.**

1. **`설계문서/` 폴더에 설계 MD를 먼저 작성한다.** (무엇을·왜·어떻게 만들지, 대안, 트레이드오프, 검증 방법)
2. **사용자와 먼저 상의한다.** 사용자가 설계에 동의하기 전에는 구현 코드를 작성하지 않는다.
3. 동의를 받은 뒤에만 구현을 시작한다.

- 이 규칙은 **모든 것**에 적용된다. "간단해 보이는 한 줄"이라도 새 기능이면 설계문서를 먼저 만든다.
- 순수 버그 수정, 오타/포맷, 사용자가 명시적으로 "바로 해줘"라고 한 경우는 상황에 맞게 판단하되, 애매하면 멈추고 물어본다.
- 설계문서 파일명 예: `설계문서/01-홈화면.md` 처럼 번호와 주제를 담는다.

**작업 원칙 (VibeNews)**
- **큰 기능 블록 하나당 설계문서 하나**를 만든다. 여러 블록을 미리 몰아서 상세 설계하지 않는다(추측성 문서 금지). 각 블록은 만들기 직전에 설계한다.
- **모든 기능은 먼저 화면 skeleton(뼈대 UI)으로 보이게** 만든다. 실제 동작은 mock/placeholder여도 된다.
- 구현이 끝나면 `README.md`와 `설계문서/README.md`를 업데이트한다.

**보안 원칙 (VibeNews)**
- `.env`, `.env.local`, API 키, DB 비밀번호, service role key 등 민감정보는 **절대 커밋하지 않는다**. `.env.example`만 커밋한다.
- 커밋/푸시 전에 `.gitignore`를 확인하고 부족하면 보강한다. **push(외부 공개)는 민감정보 검사 후 사용자 확인을 받고** 한다.
- GitHub remote: `https://github.com/leohan816/vibenews`

## 1. 코딩하기 전에 먼저 생각하라

추측하지 말라.
혼란을 숨기지 말라.
트레이드오프를 드러내라.

구현하기 전에 다음을 수행하라.

- 자신의 가정을 명확히 말하라. 확실하지 않으면 질문하라.
- 여러 해석이 가능하면 조용히 하나를 고르지 말고 선택지를 제시하라.
- 더 단순한 접근이 있다면 말하라. 필요하면 반대 의견을 내라.
- 불명확한 부분이 있으면 멈춰라. 무엇이 혼란스러운지 이름 붙이고 질문하라.

---

## 2. 단순함을 우선하라

문제를 해결하는 데 필요한 최소한의 코드만 작성하라.
추측성 코드는 쓰지 말라.

- 요청받지 않은 기능을 추가하지 말라.
- 한 번만 쓰는 코드를 위해 추상화를 만들지 말라.
- 요청받지 않은 유연성이나 설정 가능성을 추가하지 말라.
- 실제로 불가능한 상황을 위한 에러 처리를 넣지 말라.
- 50줄이면 될 일을 200줄로 썼다면 다시 줄여라.

스스로에게 물어라.

“시니어 엔지니어가 이걸 보고 과하게 복잡하다고 말할까?”

그렇다면 단순화하라.

---

## 3. 수술하듯 수정하라

반드시 필요한 부분만 건드려라.
네가 만든 문제만 정리하라.

기존 코드를 수정할 때:

- 주변 코드, 주석, 포맷을 임의로 개선하지 말라.
- 고장 나지 않은 것을 리팩터링하지 말라.
- 네 방식과 달라도 기존 스타일을 따라라.
- 관련 없는 dead code를 발견하면 삭제하지 말고 언급만 하라.

네 변경으로 인해 고아 코드가 생겼다면:

- 네 변경 때문에 사용되지 않게 된 import, variable, function은 제거하라.
- 원래부터 있던 dead code는 요청받지 않았다면 제거하지 말라.

검증 기준:

모든 변경 라인은 사용자의 요청과 직접 연결되어 있어야 한다.

---

## 4. 목표 중심으로 실행하라

성공 기준을 정의하라.
검증될 때까지 반복하라.

작업을 검증 가능한 목표로 바꿔라.

- “validation 추가” → “잘못된 입력에 대한 테스트를 작성하고 통과시킨다”
- “버그 수정” → “버그를 재현하는 테스트를 작성하고 통과시킨다”
- “X 리팩터링” → “수정 전후 테스트가 모두 통과하는지 확인한다”

여러 단계의 작업이라면 짧은 계획을 먼저 말하라.

1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]

강한 성공 기준이 있으면 독립적으로 반복할 수 있다.
약한 기준, 예를 들어 “되게 해줘” 같은 요청은 계속된 확인과 재질문이 필요하다.

---

이 지침이 잘 작동하고 있다는 신호:

- diff에서 불필요한 변경이 줄어든다.
- 과도한 복잡성 때문에 다시 쓰는 일이 줄어든다.
- 실수한 뒤 질문하는 것이 아니라, 구현 전에 명확히 질문한다.
